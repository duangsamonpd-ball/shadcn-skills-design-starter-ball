# CLAUDE.md

Guidance for Claude Code when working in this repository.

## Project Overview

A **Next.js (App Router)** application styled with **shadcn/ui + Tailwind CSS v4**, driven by a
fixed design-token system that originates in **Figma**. The Figma file is the source of truth for
design tokens (colors, spacing, radius, typography); those tokens are mirrored in the local skill
so UI can be built without guessing values.

- **Framework:** Next.js **16.2.6** (App Router, Turbopack) · React **19.2.4**
- **Styling:** Tailwind CSS v4 (`@import "tailwindcss"`, `@theme` in CSS — no `tailwind.config.js`)
- **Components:** shadcn/ui — radix base, installed via CLI into `src/components/ui/`
- **Package manager:** **npm** (do not use pnpm/yarn/bun in this repo)
- **Design source:** Figma (tokens → `DESIGN.md`; components ↔ code via Code Connect)

> ⚠️ **Next.js 16 has breaking changes vs older training data.** See `AGENTS.md` (generated by
> create-next-app) — consult `node_modules/next/dist/docs/` before using unfamiliar Next APIs.

## The Design Skill — Read This First

This repo ships a Claude Code skill at:

```
.claude/skills/shadcn-nextjs-design/
├── SKILL.md   ← rules, shadcn CLI, server/client, component & page patterns
└── DESIGN.md  ← full design-token registry (1,812 vars: colors, spacing, radius, typography)
```

The skill **auto-loads** whenever you build or edit UI (components, pages, forms, dialogs, tables,
styling). When writing any styling, **always**:

1. Read `DESIGN.md` §1–§2 for exact token values before writing color/spacing/radius classes.
2. Use **semantic tokens** (`bg-background`, `text-muted-foreground`, `border-border`) — never
   hardcoded colors (`bg-blue-500`, `text-gray-700`).
3. Install shadcn components via CLI instead of building from scratch.
4. Follow Tailwind v4 syntax (`size-4` not `w-4 h-4`; `gap-*` not `space-*`; `rounded-xs`/`shadow-xs`
   for the v3 `-sm` scale). Full v4 delta in `SKILL.md` §11.

> Invoke explicitly with `/shadcn-nextjs-design` if it does not auto-load.

## Project Structure (App Router)

```
src/
├── app/
│   ├── globals.css          ← @import "tailwindcss" + CSS variables + @theme inline
│   ├── layout.tsx           ← Providers, Toaster (root only)
│   ├── providers.tsx        ← "use client" — ThemeProvider (next-themes)
│   └── (routes)/page.tsx    ← Server Component by default
├── components/
│   ├── ui/                  ← shadcn generated — DO NOT edit directly; wrap/extend instead
│   ├── [feature]/           ← composed feature components
│   └── layout/              ← header, sidebar, footer
├── lib/utils.ts             ← cn() helper
└── hooks/                   ← custom hooks ("use client")

.figma/                      ← Code Connect mapping files (*.figma.ts) — see below
```

`globals.css` must contain the token block from `DESIGN.md` §1 verbatim (`:root`, `.dark`,
`@theme inline`). That block is the bridge between Figma tokens and Tailwind classes.

## Server vs Client Components

Default to **Server Components**. Add `"use client"` only when a file uses: `useState`/`useEffect`/
`useRef`/`useContext`, any event handler (`onClick`/`onChange`/`onSubmit`), `useTheme`, or an
interactive shadcn component (`Dialog`, `Sheet`, `DropdownMenu`, `Popover`, `Tooltip`, `Select`,
`Command`, `Tabs`). Full table in `SKILL.md` §4.

### Implementing a Figma design

1. **Read the design** — `get_design_context` (or `get_screenshot` for a visual, `get_metadata`
   for structure) on the provided Figma node URL.
2. **Pull tokens** — `get_variable_defs` to read the node's bound variables; reconcile them against
   `DESIGN.md`. If a token is missing or changed, see "Token sync" below.
3. **Build with the skill** — implement using semantic tokens + shadcn components per
   `shadcn-nextjs-design`. Never hardcode a value the design system already names.
4. **Map it back** — create/update a Code Connect file so the Figma component points to the code
   (see "Code Connect").

For composing a whole page/screen/modal into Figma from code, the `figma:figma-generate-design`
skill is the preferred path; for building/extending the design system in Figma, use
`figma:figma-generate-library`.

### Token sync (Figma variables → DESIGN.md)

`DESIGN.md` was generated from a Figma variables export (`variables-export.json`). When Figma tokens
change, refresh `DESIGN.md` and then `globals.css` §1 so code stays in sync — do not hand-edit token
values to diverge from Figma. Known Figma-source typos to correct in code are listed in
`DESIGN.md` §24 (e.g. `pimary-foreground` → `primary-foreground`, `Giest Mono` → `Geist Mono`).

## Code Connect

Code Connect maps Figma components to their code so the Figma Dev panel shows real snippets.

- Mapping files live in `.figma/` as `*.figma.ts` (one per mapped component).
- Use the `figma:figma-code-connect` skill before creating/editing these files.
- After building a reusable component (e.g. a feature `Button` wrapper or a composed card), add a
  Code Connect mapping so design and code stay linked.
- `get_code_connect_map` / `get_code_connect_suggestions` help discover existing or candidate maps.
